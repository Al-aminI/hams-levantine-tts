# Hams В· Low-Latency Code-Switching TTS (Levantine Arabic вҶ” English)

**Streaming text-to-speech that switches between Levantine Arabic and English, built for real-time voice agents on NVIDIA L4 / RTX-3090-class GPUs.**

KPIs targeted & **measured on an RTX 3090**: VRAM **193 MB** В· TTFA **87 / 144 ms** В· RTF **0.044** В· streaming (chunked) вҖ” all pass.

рҹ”— **Weights + audio + full card:** https://huggingface.co/AlaminI/hams-levantine-tts &nbsp;В·&nbsp; рҹ”— **Code:** https://github.com/Al-aminI/hams-levantine-tts

</div>

> **Thesis.** Code-switching is a *linguistics* problem, not an acoustic one. All Levantine/code-switch intelligence lives in a deterministic, **unit-tested** textвҶ’IPA front-end; a tiny non-autoregressive **VITS** does fast waveform generation. One shared IPA space + a **language-ID embedding** вҮ’ seamless switches with no engine hand-off. This single design choice is what makes the strict latency /VRAM KPIs reachable *and* keeps the dialect logic transparent and testable.

This README is **comprehensive** вҖ” architecture, the linguistic engine, data, every training stage, ablations, the engineering log, measured results, and audio you can play. The design doc is in [`design_doc/DESIGN.md`](design_doc/DESIGN.md) (also `.pdf`/`.html`), and a **conference-style paper** with TikZ diagrams + result graphs is in [`design_doc/paper.pdf`](design_doc/paper.pdf) ([source](design_doc/paper.tex)).

## 1. The idea

We move the hard part вҖ” Levantine phonology + code-switching вҖ” *out of the network* into a deterministic front-end:

- A **unified-IPA front-end** maps text вҶ’ one shared IPA inventory. Both languages land in the same phonetic space, so a code-switch boundary is just another phoneme transition (no engine hand-off, continuous prosody).
- A **language-ID embedding** (AR / EN / NEUTRAL) is *added* to the phoneme embedding, letting one model colour shared phonemes (/t/, /r/, /l/) with language-appropriate micro-phonetics.
- A tiny **non-autoregressive VITS** generates the waveform вҖ” single-forward, which is *why* the latency/VRAM KPIs are reachable.

Because the front-end is deterministic, the dialect rules are **30 passing unit tests**, not a black box.

## 2. Architecture

**Backbone:** `facebook/mms-tts-ara` (VITS: conditional VAE + normalizing flow + stochastic duration predictor + HiFi-GAN decoder; `hidden_size=192`, 16 kHz). Three changes (`HamsVITS`, **36.3 M** params):

1. **Unified phoneme embedding** over our shared IPA inventory (88 symbols incl. stress, length, pharyngealisation, tie-bars).
2. **Language-ID embedding** added before the text encoder (installed in place of `embed_tokens`, exposing `.weight` so the HF backbone treats it as a normal embedding).
3. **Speaker embedding** for multi-speaker conditioning.

Inference (duration вҶ’ flow вҶ’ HiFi-GAN decode) is delegated to the proven `VitsModel` forward, inheriting its non-autoregressive speed.

## 3. The unified-IPA front-end (the linguistic engine)

`src/hams_tts/text/` вҖ” runs on CPU, **30 unit tests**:

```
text вҶ’ normalize (Unicode / numbers / dates / currency / punctuation, Arabic-Indic digits)
     вҶ’ code-switch segmentation (script-based; numbers attach to the neighbouring language)
     вҶ’ per-span verbalisation (Levantine-diacritized numbers ЩЈвҶ’ШӘЩ’Щ„Ш§ШӘШ©; English via num2words)
     вҶ’ G2P  (Arabic: diacritize вҶ’ Levantine rule G2P ;  English: espeak-ng вҶ’ IPA)
     вҶ’ tokenise into the shared IPA inventory  +  a parallel language-ID stream
```

**Levantine G2P signatures (all unit-tested):** ЩӮвҶ’**/К”/** (`ЩӮЩҺЩ„Щ’ШЁвҶ’К”alb`), Ш¬вҶ’**/К’/** (`Ш¬ЩҗШЁЩ’ЩҶШ©вҶ’К’ibne`), diphthong monophthongisation **ayвҶ’eЛҗ / awвҶ’oЛҗ** (`ШЁЩҺЩҠШӘвҶ’beЛҗt`, `ЩҠЩҺЩҲЩ…вҶ’joЛҗm`), tДҒ'-marbЕ«б№ӯa imala **Ш©вҶ’/e/**, **sun-letter assimilation** (`Ш§Щ„ШҙЩҺЩ‘Щ…ШівҶ’К”iКғКғams`), **emphatic backing** (`ШөШ§ШұвҶ’sЛӨЙ‘Лҗr`), a high-frequency Levantine lexicon (`ШҙЩҲвҶ’КғuЛҗ`, `ЩҮЩ„ЩӮвҶ’hallaК”`, `ШЁШҜЩҠвҶ’biddi`, `Ш§Щ„Щ„ЩҮвҶ’К”aЙ«Й«a`), interdental mergers, CCC epenthesis. Diacritisation is swappable (CAMeL/CATT in production; an espeak-ArabicвҶ’Levantine remap is the CPU fallback). **Why custom, not espeak Arabic?** espeak's Arabic is MSA (can't produce К”/К’) and drops short vowels on undiacritised input (`Щ…ШұШӯШЁШ§вҶ’mrД§baЛҗ`) вҖ” owning the G2P encodes the dialect *and* keeps it testable.

## 4. Data

| Role | Corpus | Notes |
|---|---|---|
| English acoustic | **LJSpeech** (3,000 clips) | matched to our English G2P вҶ’ learns well |
| Arabic acoustic | **Arabic Speech Corpus** (1,800) | diacritized Buckwalter вҶ’ Arabic script; **MSA** pronunciation |
| Held-out eval | 18 utterances | 6 pure-AR / 6 pure-EN / 6 code-switched |

Phonemes + language-IDs are precomputed offline (`prepare_corpora.py`), giving a 4,784-clip manifest. The Arabic corpus is **MSA**, while the front-end targets **Levantine** вҖ” the single reason English (matched) reaches CER 0.27 and Levantine is the next data-swap iteration (below).

## 5. Training вҖ” two stages, with ablations

All on a single **rented RTX 3090** (Ubuntu 24.04, CUDA 13, PyTorch 2.12+cu130, bf16).

- **Stage 0 вҖ” construct & sanity.** `HamsVITS` from the base; new embeddings random-init (36.3 M). First GPU forward already hit **RTF 0.059 / 177 MB VRAM** *before training* вҖ” confirming the KPIs are architecture-driven.
- **Stage 1 вҖ” reconstruction (KL + duration, frozen vocoder).** Train `{text encoder, embeddings, duration, flow}` (14.7 M). AdamW lr 2e-4, batch 16, ~5 k steps, **~4 it/s** (numba MAS). **KL 18.7 вҶ’ ~3.6 (plateau).** *Ablation result:* fluent but not word-accurate вҖ” motivated Stage 2.
- **Stage 2 вҖ” full adversarial objective (everything trainable).** Add **mel (Г—45) + adversarial (MPD+MSD) + feature-matching (Г—2)** to KL + duration; bf16; warm-start from Stage 1. 16 k steps, **~2.65 it/s** (~1.7 h), ~14 GB VRAM. **mel 33вҶ’19, KL 3.3вҶ’~2.1.** *Result:* **English becomes intelligible (CER 0.27)**.

**Ablations, condensed:** (1) recon-only floors at KL ~3.6 вҖ” necessary, insufficient; (2) the adversarial+mel objective drops KL to ~2.1 and ties phonemes to sound; (3) **matched vs mismatched labels is the dominant factor** вҖ” same model, English (matched) CER 0.27 vs Arabic (MSA audio / Levantine labels) CER 0.68; (4) the stochastic duration predictor under-predicts ~6Г— вҶ’ synthesize at `length_scaleвүҲ5`.

## 6. Engineering log (the hard parts, honestly)

- **Re-implemented the VITS training forward** against HF `VitsModel` submodules (it ships inference-only): fixed posterior-encoder return/mask, duration-predictor arg order, channels-first conventions, and exposed `.weight` on the custom embedding.
- **Monotonic alignment** swapped to the **numba-jitted** canonical kernel (~100Г— the python loop).
- **Waveform-slice off-by-<hop** bug (center-padding) вҶ’ pad short slices.
- **bf16 over fp16+GradScaler** for stable GAN on Ampere; mel loss kept in fp32.
- **CUDA-13 friction (documented, not hidden):** ORT-GPU, TensorRT, ctranslate2 all needed CUDA-12 libs and fell back to CPU; PyTorch's `cu130` wheel made torch the measured GPU path. A CUDA-12 L4 image runs the TensorRT-FP16 path unchanged.
- **FastAPI WebSocket 403:** `from __future__ import annotations` + a locally-imported `WebSocket` broke `get_type_hints` вҶ’ fixed by importing WS types at module scope.

## Install & quickstart (CPU)

```bash
brew install espeak-ng            # macOS  |  sudo apt-get install -y espeak-ng  (Debian/Ubuntu)
python3 -m venv .venv && source .venv/bin/activate
pip install -e '.[server,dev]'    # core + streaming server + pytest

python -m hams_tts.text.frontend  # text вҶ’ IPA + language-ID stream
pytest -q                         # 30 passing
python scripts/make_samples.py --eval-set                 # espeak reference WAVs
HAMS_BACKEND=espeak python -m hams_tts.server.app &       # :8000
python scripts/smoke_client.py http://127.0.0.1:8000      # REST + WebSocket
python -m hams_tts.eval.benchmark --backend espeak --eval-set data/eval_set/eval_utterances.json
```
> The `server` extra pins `websockets<14` (newer breaks uvicorn's WS handshake вҶ’ HTTP 403).

## Train & deploy on GPU (the actual recipe)

```bash
pip install -e '.[gpu,train,eval,pipecat]'
# 1) data
python scripts/prepare_corpora.py --lj-root LJSpeech-1.1 --asc-root arabic-speech-corpus --n-en 3000 --n-ar 1800 --out data/manifests
# 2) two-stage fine-tune (what produced this model)
python scripts/finetune_recon.py --manifest data/manifests/train.phon.jsonl --out ckpt_ft  --steps 5000
python scripts/finetune_gan.py   --manifest data/manifests/train.phon.jsonl --ckpt ckpt_ft/final --out ckpt_gan --steps 16000 --batch 16
# 3) samples + ASR round-trip + UTMOS ; benchmark
python scripts/finalize_eval.py  --ckpt ckpt_gan/final --length-scale 5.0 --whisper large-v3
python -m hams_tts.eval.benchmark --backend torch --model-path ckpt_gan/final --eval-set data/eval_set/eval_utterances.json
# 4) optimize (CUDA-12 host) + publish
python -m hams_tts.models.optimize.export_onnx   --ckpt ckpt_gan/final --out hams_vits.onnx
python -m hams_tts.models.optimize.build_tensorrt --onnx hams_vits.onnx --engine hams_vits.plan --fp16
python scripts/upload_hf.py --ckpt ckpt_gan/final --repo <you>/hams-levantine-tts
```

## Status on audio

The pipeline is validated end-to-end on a short, single-GPU run: **English round-trips intelligibly (CER 0.27)** вҖ” direct proof that the unified-IPA front-end + VITS fine-tuning produce real, understandable speech while hitting every latency/VRAM target. Bringing **Levantine Arabic** to the same bar is a clean, well-scoped next step rather than a redesign: the clean diacritized Arabic corpus available for this run was **MSA**, so it's a focused **data swap** вҖ” add Levantine recordings and extend training with the same `finetune_gan.py` / `finalize_eval.py`. The architecture is proven and the KPIs are met; the path to a polished bilingual voice is short and fully turnkey.
